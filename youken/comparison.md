| 目的 | 3つのアプローチを比較し、本システムに最適な残業申請方式を選定する |

---

## 1. 3案の概要

| 項目 | 案A: 厳密型 | 案B: バランス型 | 案C: 柔軟型 |
|------|-----------|---------------|-----------|
| 基本方針 | 事前申請必須、例外なし | 事前申請が原則、上長承認で事後も可 | 事後申請をメインとする |
| 一言で表すと | 「申請なくして残業なし」 | 「原則事前、現場判断で事後も認める」 | 「まず働き、後から申請」 |

## 2. 各案の詳細仕様

### 案A: 厳密型（事前申請必須、例外なし）

#### ワークフロー

```
従業員が残業申請（開始前）
  → 上長が承認
    → 承認済みの場合のみ、所定終業時刻以降の勤務を残業として記録
      → 未承認の状態で所定終業時刻を超過 → システムが警告表示＋強制ログアウト通知
```

#### 要件定義

| ID | 要件 |
|----|------|
| A-006 | 残業は必ず事前申請を行い、上長の承認を得た上で実施すること。事後申請は認めない |
| A-007 | 残業申請は所定終業時刻の1時間前までに提出すること |
| A-008 | 残業申請には予定残業時間（開始〜終了）と業務内容・理由を記入すること |
| A-009 | 上長は残業申請を承認・差戻し・却下できること。申請から30分以内に対応がない場合、上長にリマインド通知を送ること |
| A-010 | 承認済み残業の予定終了時刻を超過した場合、延長申請を必須とすること |
| A-011 | 残業申請なしに所定終業時刻を15分以上超過した退勤が記録された場合、本人にコンプライアンス違反として通知し、上長にも報告すること |

### 案B: バランス型（事前原則、上長承認で事後可）

#### ワークフロー

```
【通常フロー】
従業員が残業申請（事前）
  → 上長が承認
    → 残業実施・退勤打刻

【緊急フロー】
緊急の残業が発生（顧客トラブル・締切等）
  → 残業を実施・退勤打刻
    → 翌営業日始業までに従業員が事後申請（理由必須）
      → 上長が承認（事後申請の妥当性を判断）
        → 承認: 正式な残業として記録
        → 却下: サービス残業の疑いとして人事部に自動報告
```

#### 要件定義

| ID | 要件 |
|----|------|
| B-006 | 残業は事前申請を原則とし、従業員が申請 → 上長が承認のワークフローで運用すること |
| B-007 | 緊急時は事後申請を認めること。事後申請の期限は当該勤務日の翌営業日の始業時刻までとし、理由の記入および緊急事由の選択（顧客対応/システム障害/締切対応/その他）を必須とすること |
| B-008 | 残業申請には予定残業時間（開始〜終了）と理由を記入できること。事後申請の場合は実績時間を記入すること |
| B-009 | 上長は残業申請を承認・差戻し・却下できること。差戻し・却下時はコメントを必須とすること |
| B-010 | 承認されていない残業が発生した場合、退勤打刻時に事後申請画面を自動表示し、翌営業日の始業時刻までに提出を促す通知を送ること |
| B-011 | 事後申請率が部署内で月間30%を超えた場合、人事部にアラートを送ること |
| B-012 | 事後申請の却下が行われた場合、当該残業時間をサービス残業の疑いとして人事部に自動報告すること |

### 案C: 柔軟型（事後申請メイン）

#### ワークフロー

```
従業員が残業を実施・退勤打刻
  → システムが所定終業時刻との差分を自動計算
    → 残業が発生していた場合、退勤打刻時に申請画面を自動表示
      → 従業員が理由を記入して申請
        → 上長が翌営業日中に一括承認
```

#### 要件定義

| ID | 要件 |
|----|------|
| C-006 | 残業申請は事後申請を標準フローとする。事前申請も可能だが必須としない |
| C-007 | 退勤打刻時に所定終業時刻を超過していた場合、残業申請画面を自動表示し、理由の記入を促すこと |
| C-008 | 残業申請には実績残業時間（自動計算）と理由を記入すること。従業員は実績時間を手動修正できないこと（打刻データから自動算出） |
| C-009 | 上長は未承認の残業申請を一覧で確認し、一括承認・個別却下ができること |
| C-010 | 退勤打刻後48時間以内に申請が未提出の場合、自動リマインドを送信すること。72時間経過で上長に未申請アラートを送ること |
| C-011 | 月間の残業実績を自動集計し、申請漏れがないことを月末に本人・上長に確認通知を送ること |

## 3. 比較表

### 3.1 メリット・デメリット比較

| 評価項目 | 案A: 厳密型 | 案B: バランス型 | 案C: 柔軟型 |
|---------|-----------|---------------|-----------|
| **メリット1** | 残業の完全な事前統制により、不要な残業を抑制できる | 原則を維持しつつ現場の実態に対応でき、最も現実的な運用が可能 | 従業員の申請負荷が最も低く、打刻データとの乖離が起きにくい |
| **メリット2** | 労基署対応の際、全残業が事前承認済みであることを証明できる | 事後申請の却下→サービス残業検知の仕組みにより、労務リスクを可視化できる | 退勤打刻と申請が同時のため、申請忘れが起きにくい |
| **メリット3** | 管理者が残業実態を事前に把握でき、業務配分の最適化がしやすい | 事後申請率のモニタリングにより、運用の健全性を定量的に監視できる | 上長は一括承認できるため、承認作業の負荷が低い |
| **デメリット1** | 突発対応（顧客クレーム、システム障害等）で申請が間に合わない場面が必ず発生する | 「緊急」の定義が曖昧になりやすく、事後申請が常態化するリスクがある | 残業の事前統制が効かず、不要な残業の抑制力が弱い |

## 4. 推奨案

### 第一推奨: 案B（バランス型）

案Bを推奨する理由は以下の3点です。

**1. サービス残業の防止と法令遵守の両立**

案Aは制度上は最も厳格だが、実運用では「申請が間に合わないから申請せずに残業する」というサービス残業の潜在化リスクが最も高い。案Bは事後申請を公式に認めることで残業の実態を捕捉しやすくし、却下時にはサービス残業疑いとして人事部に自動報告する仕組み（B-012）により、労務リスクを可視化できる。

**2. 運用の持続可能性**

案Aは上長に過大な承認負荷がかかり、形骸化して結果的に案Cに近い運用に崩れるケースが多い。案Bは「事前原則」を維持しつつ現場の裁量を認めるため、制度と実態の乖離が起きにくく、長期的に持続可能な運用である。

**3. 事後申請率による自律的な改善サイクル**

案B独自の「事後申請率30%超過アラート」（B-011）により、部署ごとの運用健全性を定量的にモニタリングできる。事後申請率が高い部署は業務改善や人員配置の見直し対象として可視化されるため、組織全体の残業管理が自律的に改善される仕組みを内包している。

### requirement_v2.md への影響

案Bを採用する場合、現行のREQ-006〜REQ-010は案Bの方針と概ね整合しているため、以下の差分対応が必要です。

| 対応 | 内容 |
|------|------|
| REQ-007 修正 | 事後申請時に「緊急事由の選択」を追加（B-007） |
| REQ-008 修正 | 事後申請の場合は「実績時間」を記入する旨を追加（B-008） |
| REQ-042 新規 | 事後申請率の月次モニタリングと閾値アラート（B-011） |
| REQ-043 新規 | 事後申請の却下時、サービス残業の疑いとして人事部に自動報告（B-012） |
